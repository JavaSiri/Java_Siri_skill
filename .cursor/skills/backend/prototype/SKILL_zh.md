# Prototype

原型是**用来回答问题的可丢弃代码**。问题本身决定了原型的形态。

## 选一条分支

明确当前要回答的问题是什么 —— 从用户的提示、周边代码中判断，或直接询问用户：

- **"这个逻辑 / 状态模型是否合理？"** → [LOGIC.md](LOGIC.md)。构建一个小型的交互式终端应用，将状态机推入难以在纸上推导的边界场景。
- **"这个应该长什么样？"** → [UI.md](UI.md)。在同一路由下生成多个风格迥异的 UI 方案，通过 URL 查询参数和浮动底部栏切换。

两条分支产出的东西完全不同 —— 选错分支会浪费整个原型。如果问题本身确实模糊，且无法联系到用户，默认选择与周围代码更匹配的分支（后端模块 → logic；页面或组件 → UI），并在原型顶部说明这一假设。

## 两条分支通用的规则

1. **从第一天起就标记为可丢弃。** 将原型代码放在它实际对应的模块或页面附近，上下文一目了然 —— 但命名要让普通读者一眼看出这是原型而非生产代码。对于可丢弃的 UI 路由，遵循项目现有的路由规范；不要另起一套顶层结构。
2. **一条命令即可运行。** 使用项目现有的任务运行器 —— `pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能在不假思索的情况下启动它。
3. **默认不持久化。** 状态存在于内存中。持久化是需要原型去**验证**的东西，而非原型依赖的东西。如果问题明确涉及数据库，使用临时数据库或本地文件，并明确命名为"PROTOTYPE — 随时删除"。
4. **跳过打磨。** 不写测试，除了让原型**能跑起来**之外的错误处理一概不做，不做抽象。目的是快速学习，然后删掉。
5. **暴露状态。** 每次操作后（logic）或每次切换方案时（UI），打印或渲染完整的相关状态，让用户看到变化。
6. **完成后删除或吸收。** 原型回答了问题之后，要么删除它，要么将经过验证的决策合并到真实代码中 —— 不要让它在仓库里腐烂。

## 完成后

原型**唯一的价值在于答案**。将答案记录到持久的地方（commit message、ADR、issue，或原型旁的 `NOTES.md`），并附带它所回答的问题。如果用户在身边，记录这个过程就是一次快速对话；如果不在，留下占位符让他们（或下次迭代时的你）在删除原型前填入结论。
