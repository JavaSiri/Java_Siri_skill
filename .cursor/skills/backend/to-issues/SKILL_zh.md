# To Issues

使用垂直切片（tracer bullets）方法将计划拆解为独立可领取的 issue。

issue 追踪系统和标签词汇表应该已经配置好——如果没有，运行 `/setup-matt-pocock-skills`。

## 工作流程

### 1. 收集上下文

基于对话中已有的内容工作。如果用户传递了 issue 引用（issue 编号、URL 或路径）作为参数，从 issue 追踪系统拉取并阅读其完整内容和评论。

### 2. 探索代码库（可选）

如果尚未探索过代码库，先了解当前代码状态。issue 标题和描述应使用项目的领域术语词汇，并遵循所涉及区域的 ADR。

### 3. 起草垂直切片

将计划拆解为** tracer bullet** issue。每个 issue 是一个薄的垂直切片，贯穿**所有**集成层级端到端，**而非**某一层的水平切片。

切片可以是 "HITL" 或 "AFK"。HITL 切片需要人工交互，如架构决策或设计评审。AFK 切片可以无需人工交互即可实现和合并。优先选择 AFK 而非 HITL。

<vertical-slice-rules>
- 每个切片在每一层（schema、API、UI、tests）都交付一条窄而**完整**的路径
- 一个完成的切片可独立演示或验证
- 优先选择多个薄切片，而非少量厚切片
</vertical-slice-rules>

### 4. 询问用户

以编号列表展示切片方案。每个切片包含：

- **标题**：简短的描述性名称
- **类型**：HITL / AFK
- **阻塞于**：哪些其他切片（如有）必须先完成
- **覆盖的用户故事**：此切片对应哪些用户故事（如果源材料中有的话）

询问用户：

- 粒度是否合适？（太粗 / 太细）
- 依赖关系是否正确？
- 是否需要合并或进一步拆分？
- HITL 和 AFK 标注是否正确？

迭代直到用户批准切片方案。

### 5. 将 issue 发布到 issue 追踪系统

对每个已批准的切片，在 issue 追踪系统中发布一个新 issue。使用下方的 issue 正文模板。这些 issue 被视为已准备好供 AFK agent 处理，因此除非另有指示，否则应带上正确的 triage 标签发布。

按依赖顺序（阻塞项在前）发布 issue，以便在 "Blocked by" 字段中引用真实的 issue 标识符。

<issue-template>
## 父级

对 issue 追踪系统中父级 issue 的引用（如果源是已有 issue，否则省略此部分）。

## 做什么

对这个垂直切片的简洁描述。描述端到端行为，而非逐层实现。

避免具体的文件路径或代码片段——它们很快就会过时。例外：如果原型产生了一个片段，其编码的决策比文字描述更精确（如状态机、reducer、schema、类型定义），可将其内联于此，并简要说明它来自原型。精简到决策密集的部分——不是可工作的 demo，只是重要的部分。
