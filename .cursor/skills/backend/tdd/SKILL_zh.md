# 测试驱动开发

## 理念

**核心原则**：测试应通过公共接口验证行为，而非实现细节。代码可以彻底重构；测试不应随之崩溃。

**好的测试**是集成式的：它们通过公共 API 走真实的代码路径。测试描述的是系统**做什么**，而不是**怎么做**。一条好的测试读起来像规格说明 —— "用户可以用有效购物车结账"精确告诉你存在什么能力。这类测试能在重构中存活，因为它们不关心内部结构。

**差的测试**与实现耦合。它们 mock 内部协作方、测试私有方法、或通过外部手段验证（如直接查询数据库而非使用接口）。警示信号：重构后测试就挂了，但行为并没有变。如果重命名一个内部函数后测试失败，那说明测试测的是实现而非行为。

相关示例见 [tests.md](tests.md)，mock 指南见 [mocking.md](mocking.md)。

## 反模式：水平切分

**不要先写完所有测试，再写所有实现。** 这叫"水平切分"—— 把 RED 阶段当成"写完所有测试"，GREEN 阶段当成"写完所有代码"。

这会产出**垃圾测试**：

- 批量写出的测试测的是**想象中的**行为，而非**真实的**行为
- 最终测的是事物的**形状**（数据结构、函数签名），而非用户可见的行为
- 测试对真实变化失去敏感 —— 行为坏了它通过，行为正常它失败
- 超越了自己的视线范围，在还没理解实现之前就锁定了测试结构

**正确做法**：通过追踪弹（tracer bullet）做垂直切分。一次测试 → 一次实现 → 循环。每个测试都回应了前一轮学到的东西。因为刚写完代码，所以清楚地知道什么行为重要、怎么验证它。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域词汇表，使测试名称和接口词汇与项目语言一致，并尊重所在区域的 ADR。

在写任何代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测试哪些行为（排优先级）
- [ ] 识别[深度模块](deep-modules.md)的机会（小接口、深实现）
- [ ] 为[可测试性](interface-design.md)设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的批准

提问："公共接口应该长什么样？哪些行为最重要、最需要测试？"

**你不可能测完所有东西。** 与用户明确哪些行为最重要。把测试精力集中在关键路径和复杂逻辑上，而不是每个可能的边界情况。

### 2. 追踪弹

写**一个**测试，确认系统**一件事**：

```
RED:   写第一个行为的测试 → 测试失败
GREEN: 写最少代码让测试通过 → 测试通过
```

这就是追踪弹 —— 证明了这条路是通的。

### 3. 增量循环

对每个剩余的行为：

```
RED:   写下个测试 → 失败
GREEN: 最少代码让测试通过 → 通过
```

规则：

- 一次只写一个测试
- 只写足够通过当前测试的代码
- 不为未来测试预留
- 保持测试聚焦于可观察行为

### 4. 重构

所有测试通过后，寻找[可重构候选](refactoring.md)：

- [ ] 提取重复代码
- [ ] 加深模块（将复杂性藏在简单接口背后）
- [ ] 在自然之处应用 SOLID 原则
- [ ] 考虑新代码揭示了关于现有代码的什么
- [ ] 每次重构后运行测试

**红的时候绝不重构。** 先到绿。

## 每轮检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只使用公共接口
[ ] 测试在内部重构后能存活
[ ] 代码对这个测试而言是最少的
[ ] 没有添加投机性功能
```
