# 不予处理知识库

项目的 `.out-of-scope/` 目录用于持久记录被拒绝的需求。服务两个目的：

1. **制度记忆** — 记录为什么拒绝该需求，避免 reasoning 随 issue 关闭而丢失
2. **去重** — 新 issue 来时匹配历史拒绝记录，无需重复辩论

## 目录结构

```
.out-of-scope/
├── daily-sku-stock-rollup.md
├── real-time-ads-layer.md
└── user-profile-360.md
```

一个文件对应一个**概念**，而非一个 issue。多个 issue 请求同一件事归入一个文件。

## 文件格式

文件应采用轻松、可读的写作风格——更像一份简短的设计文档，而非数据库条目。使用段落、代码示例和示例让 reasoning 对首次接触的人清晰有用。

```markdown
# 历史 SKU 库存全量快照

该项目不提供商品历史库存快照功能。

## 为什么不在范围内

ODS 层的设计目标是承接增量数据变更（CDC 格式），而非保留全量历史状态。提供全量快照会导致：

- ODS 表数据量呈线性增长，历史分区无限膨胀
- 与下游 DWD 的拉链表设计冲突，产生数据口径歧义
- 与现有 `dwd_inventory_daily_snapshot` 拉链表功能重叠

历史库存分析应通过拉链表（`dwd_inventory_daily_snapshot`）实现，而非新增全量快照表。

## 先前请求

- #42 — "新增商品历史库存快照表"
- #87 — "支持查看任意时间点商品可用库存"
- #134 — "SKU 库存回溯功能"
```

### 文件命名

使用简短、描述性的 kebab-case 名称：`real-time-ads-layer.md`、`cross-db-join.md`。名称应足够清晰，让浏览目录的人无需打开文件就能理解被拒绝的概念。

### 撰写原因

原因要实质性——不是"我们不想要这个"，而是要有为什么。好的原因应引用：

- 项目范围或原则（"该项目聚焦于离线批处理；实时需求属于另一套架构范畴"）
- 技术约束（"实现这一功能需要打通 ODS 和 DWD 的历史分区，涉及大量 ETL 重写"）
- 战略决策（"我们选择使用拉链表而非快照表，因为……"）
- 数仓分层策略（"ADS 层仅供查询，不做数据生产；该需求需要生产写入，违反分层原则"）

原因应持久。避免引用临时情况（"我们目前太忙了"）——那不是真正的拒绝，只是延期。

## 何时检查 `.out-of-scope/`

在分类的第 1 步（收集上下文）中，读取 `.out-of-scope/` 下所有文件。当评估新 issue 时：

- 检查请求是否匹配已有的 out-of-scope 概念
- 匹配按概念相似性，而非关键词匹配——"商品历史库存"匹配 `daily-sku-stock-rollup.md`
- 如果有匹配，向维护者呈现：`"这个需求与 `.out-of-scope/daily-sku-stock-rollup.md` 相似——我们之前拒绝了这个，原因是【reason】。你仍然认为应该这样处理吗？"`

维护者可以：

- **确认** — 将新 issue 添加到现有文件的"先前请求"列表，然后关闭
- **重新考虑** — 删除或更新 out-of-scope 文件，issue 按正常流程继续
- **不同意** — 两个 issue 相关但有区别，按正常流程继续

## 何时写入 `.out-of-scope/`

仅当一个 **Enhancement**（而非 Bug）被标记为 `wontfix` 时。流程：

1. 维护者决定某个功能需求不在范围内
2. 检查是否已有匹配的 `.out-of-scope/` 文件
3. 如果有：在其"先前请求"列表追加新 issue
4. 如果没有：创建新文件，包含概念名称、决策、原因和第一个先前请求
5. 在 issue 下发布评论，说明决策并提及 `.out-of-scope/` 文件
6. 用 `wontfix` 标签关闭 issue

## 更新或移除 out-of-scope 文件

如果维护者对之前拒绝的概念改变了想法：

- 删除 `.out-of-scope/` 文件
- 无需重新打开旧的 issue——它们是历史记录
- 触发本次重新考虑的新 issue 按正常流程继续分类

## 数据仓库场景下的特殊考量

与纯代码项目不同，数据仓库的 `.out-of-scope/` 决策通常涉及：

- **分层策略** — 该需求是否违反 ODS/DWD/DWS/ADS 的分层原则
- **数据一致性** — 该需求是否与其他表的口径存在冲突
- **性能与成本** — 全量快照、历史分区无限膨胀等资源问题
- **架构一致性** — 离线批处理 vs 实时流处理的技术栈边界
- **上游依赖** — 该需求是否依赖不可用的上游数据源
