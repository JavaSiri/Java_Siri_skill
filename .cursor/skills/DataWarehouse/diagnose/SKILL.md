# Diagnose

数据仓库疑难问题的规范化诊断流程。只在有明确理由时才跳过某个阶段。

## 阶段 1 — 建立反馈循环

**这才是技能本身。** 其他一切都是机械操作。如果有一个快速的、确定性的、可重复的 SQL 执行信号，你就能找到原因——逐层追查、假设验证、执行计划分析都只是在消耗那个信号。如果没有信号，盯再久代码也找不出问题。

**在这里投入不成比例的努力。要激进，要创造性，拒绝放弃。**

### 构建反馈循环的方式——大致按此顺序尝试

1. **SQL 直接执行**——在触及问题的任意层级执行：ODS / DWD / DWS / ADS 层查询，验证数据是否异常。
2. **CLI 查询**——`beeline` / `hive -e` / `psql -c` / JDBC 连接，对比已知正确的结果快照。
3. **调度日志排查**——查看 Airflow / DolphinScheduler / 自研调度平台的任务日志，确认任务是否成功执行。
4. **跨环境对比**——Dev vs Prod 执行同一 SQL，对比结果差异。
5. **逐层溯源**——ADS 异常 -> DWS -> DWD -> ODS，层层追查上游数据。
6. **分区重跑验证**——重跑当日分区（`INSERT OVERWRITE`），观察问题是否复现。
7. **数据量采样**——`SELECT COUNT(*)`, `SELECT COUNT(col)`, `SELECT COUNT(DISTINCT col)`，定位数据量级异常。
8. **数据飘移检测**——对比历史同期数据增长率、空值率、Schema 变更。
9. **HITL 手动脚本**——最后手段。如果必须人工执行 SQL / 查调度平台 / 看分区状态，用结构化脚本驱动（`scripts/hitl-dw-loop.sh`），将输出截图或 `KEY=VALUE` 结果反馈给你。

构建好正确的反馈循环，问题就解决了 90%。

### 对循环本身做迭代

把循环当作一个产品。一旦你有了*一个*循环，问自己：

- 能让它更快吗？（只查当日分区、限制返回行数、跳过无关的初始化步骤。）
- 能让信号更锐利吗？（对具体症状做断言，而不是"好像少了点数据"。）
- 能让它更确定性吗？（固定日期分区、排除并发写入干扰、隔离时区影响。）

一个 30 秒的不稳定循环比没有循环好不了多少。一个 2 秒的确定性循环是诊断利器。

### 非确定性数据问题

目标不是干净的重现，而是**更高的复现率**。多查几个分区、扩大时间窗口、增加 JOIN 条件提高触发概率。A/B 测试同一上游数据、对比多个任务批次的结果。50% 能复现的问题可诊断；1% 几乎无法诊断——不断提高复现率直到可诊断为止。

### 当你真的无法构建循环时

停下来，明确说出来。列出你尝试过的方式。向用户请求：(a) 访问能复现问题的生产或测试环境，(b) 捕获的产物（任务日志、数据导出、截图、时间戳），或 (c) 上下游表结构的 DDL。**在没有循环的情况下不要进入假设阶段。**

在确认有一个你信任的循环之前，不要进入阶段 2。

## 阶段 2 — 复现

运行反馈循环，观察数据问题出现。

确认：

- [ ] 循环产生的异常是用户**描述的**那个——不是碰巧在附近的一个不同问题。找错问题 = 修错地方。
- [ ] 问题在多个分区 / 多次运行中可复现（或者以足够高的频率可复现以支持诊断）。
- [ ] 你已捕获了确切症状（错误数据、缺失分区、指标偏差值），这样后续阶段可以验证修复是否真正解决了问题。

在复现问题之前不要继续。

---

## 阶段 3 — 假设

在测试任何一个之前，生成 **3–5 个排序后的假设**。只生成单个假设会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它所做的预测。

> 格式："如果 X 是原因，那么改变 Y 会让问题消失 / 改变 Z 会让问题更严重。"

如果无法陈述预测，这个假设只是"感觉"——丢弃它或把它磨利。

**常见 DW 假设方向：**

- 上游 ODS 缺失分区或数据延迟，导致下游任务未产出
- 字段空值 / 脏数据导致聚合结果异常（如 `SUM(NULL)` = 0 被错误计入）
- 日期分区条件写死为特定日期，遇到跨月/跨年数据截断
- 时区问题导致日期边界数据错位（如 UTC vs 北京时间）
- 任务调度依赖配置错误，上游任务未完成就触发下游
- 数据飘移（Schema 漂移：上游新增字段但下游未处理；空值率异常升高）
- 重复数据导致指标翻倍或去重逻辑失效
- JOIN 键包含 NULL 导致的数据丢失
- 数据倾斜（skew join）导致部分 reducer OOM 或结果错误
- 窗口函数边界写错（如 `ROWS BETWEEN` vs `RANGE BETWEEN`）

**在测试之前把排序后的列表展示给用户。** 他们通常有业务知识能立即重新排序（"这个字段上周刚改名了，所以 #3 最可能"），或者知道你已经排除掉的假设。这是一个廉价的检查点，节省大量时间。不要在这里卡住——如果用户不在，继续按你的排序进行。

---

## 阶段 4 — 插桩

每个探测必须对应阶段 3 中的一个具体预测。**一次只改变一个变量。**

工具偏好：

1. **`EXPLAIN` / `EXPLAIN ANALYZE`**——PostgreSQL / Hive 执行计划分析，定位 SQL 逻辑错误或性能瓶颈。
2. **数据采样验证**——`SELECT * LIMIT 100` 抽查原始数据，验证假设是否与实际数据吻合。
3. **对比验证**——分别对正常分区和问题分区执行同一查询，直观对比差异。
4. **元数据审计**——`DESC FORMATTED table` / `SHOW PARTITIONS` / `information_schema` 确认分区状态和 Schema 是否一致。
5. 永远不要"查所有历史分区然后肉眼找不同"。

**给每次诊断查询打上注释或标签**，例如 `-- [DIAG-01] 检查空值率`。汇总所有诊断 SQL 到文件末尾，便于一次性清理或复用。

**性能问题分支。** 如果问题是延迟或慢查询，日志通常不够用。替代方案：先用 `EXPLAIN` 建立基线，理解执行计划；然后逐个移除 JOIN / 子查询定位瓶颈；最后针对瓶颈加索引 / 改写 SQL。先测量，再修复。

---

## 阶段 5 — 修复 + 回归测试

**在修复之前先写 UAT 验证 SQL**——但仅当存在**正确的验证边界**时。

正确的验证边界是指验证 SQL 在调用点真实地复现了问题的实际模式。如果验证 SQL 太浅（比如只查了行数但问题出在某个聚合指标的精度），那这个验证给的是虚假的信心。

**如果不存在正确的验证边界，这本身就是一个发现。** 记录它。表结构或调度设计正在阻止问题被可靠地验证。标记到下一阶段。

如果存在正确的验证边界：

1. 将最小化的复现转化为在该边界处的一条 UAT 验证 SQL。
2. 执行它，确认问题存在。
3. 应用修复（改 SQL / 重刷分区 / 调整调度依赖）。
4. 重新执行 UAT 验证 SQL，确认问题消失。
5. 对比修复前后的关键指标（行数、聚合值），确保数量级一致。
6. 用原始（未最小化的）场景重新运行阶段 1 的反馈循环。

---

## 阶段 6 — 清理 + 复盘

在宣布完成之前必须完成：

- [ ] 原始问题不再复现（重新运行阶段 1 循环）
- [ ] UAT 验证 SQL 通过（或记录了不存在验证边界的事实）
- [ ] 所有诊断 SQL 清理或标注（`grep` 搜索 `[DIAG-` 前缀）
- [ ] 一次性调试表 / 临时分区已清理
- [ ] 正确的假设和修复措施记录在 Wiki 或数据字典中——以便下次同类问题快速定位

**然后问：什么能预防这个问题？** 如果答案涉及架构变更（ODS 层缺少审计字段、DWD 层缺少数据质量校验、调度缺少依赖检查），建议补充相应机制。**在修复完成之后**而不是之前提建议——你现在比开始时拥有更多信息。
