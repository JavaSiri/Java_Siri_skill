# table-requirement-analyzer

## 强制确认门

> **AI 绝对禁止跳过本 skill。** 无论用户是否认为需求已清晰、无论 AI 主观判断是否"够明确"，都必须执行本 skill 的追问流程，向用户逐维度发出确认提示，等待用户响应后才能推进。

跳过本 skill 即违反工作流纪律，视为未完成需求澄清。

---

## Step 0: 读取知识库（AI 主动判断）

> 在抛出任何确认问题前，**必须先读取以下知识资产**，建立对现有体系的理解，AI 才能作出有依据的判断和追问。

```
必须读取的知识库（按顺序）：
1. .cursor/knowledge/business/entity-relationship.md     ← 业务主体与关系模型
2. .cursor/knowledge/schema/ 目录                         ← 现有表结构与字段定义
3. .cursor/knowledge/metrics/ 目录                       ← 已有指标口径（如有）
4. .cursor/knowledge/requirements/ 目录                   ← 历史表需求卡（参考格式）
```

**AI 在此步须自行判断并输出以下结论**（不依赖用户告知）：

```
【AI 先行判断结论】
- 目标表是否已存在：{存在 / 不存在}
- 若存在，现有字段：{列表}
- 若不存在，同主题域是否有可参考的表：{表名列表}
- 目标表与现有体系的关系：{独立新建 / 依赖已有表 / 扩展已有表}
- 数据来源是否已有 ODS 层支撑：{是 / 否}
- 是否需要触发 ODS 入湖前置任务：{是 / 否}
```

**AI 不得在未完成上述判断前向用户提问。**

---

在抛出任何确认问题前，必须先回答三个问题：

1. 当前正在处理哪个需求？（回答目标表名 + 操作类型）
2. 需求是否已全部确认？（回答 "pending" 或 "confirmed"）
3. 上一个维度是否已确认？未确认不得进入下一个

这三个问题回答后，方可发送追问消息。

## 追问纪律

- **AI 判断先于追问**：所有追问必须基于 Step 0 读取知识库后的判断结论，不得脱离已有知识凭空提问。追问的目的是"用户告知 vs AI 已掌握"的差异核对，而非从头解释。
- **禁止自主判断需求清晰**：无论 AI 认为多么清晰，都必须向用户发出确认提示，等待用户确认后方可标记为已澄清。AI 无权单方面跳过此步骤。
- **一次只问一个问题**：每次只抛出一个歧义点，等待用户回答后再问下一个。禁止一次性抛出多个问题。
- 必须在当前维度的需求完全澄清后，将追问记录追加写入工作上下文，方可进入下一个追问。
- **追问措辞参考**：以 "我理解的是……，是这样吗？" 或 "现有知识库中已有……，这次的需求是否沿用？" 的方式提问，保持高效确认而非从头解释。

## 澄清结论记录

每澄清一个维度后，将该维度的追问问答追加写入工作上下文（不写独立文件），格式：

```
【维度 N 澄清记录】
Q: {AI 追问}
A: {用户回答}
结论: {确认事项}
```

维度追问须**即时写入上下文**，不允许跨消息累积后统一补写。

---

## 澄清维度（AI 动态生成）

澄清维度不是固定模板，而是 AI 根据 Step 0 知识库分析结果，动态识别出的具体歧义点。

**AI 应当识别但不限于以下维度类型**：

- **业务定位歧义**：粒度定义不清晰、主体关系模糊
- **字段定义歧义**：字段缺失、类型不匹配、与参考表不一致
- **存储策略歧义**：分区方案、生命周期未明确
- **ETL 逻辑歧义**：来源不明确、增量/全量未确认、清洗规则缺失
- **下游依赖歧义**：消费方未确认、元数据未同步

**AI 动态生成追问的逻辑**：

1. 从 Step 0 读取知识库后，对比"用户需求"与"现有体系"的差异
2. 差异点即歧义点，每个歧义点生成一个追问
3. 追问必须具体到"差异是什么"，而非泛泛提问

**示例**（比固定模板更有效的追问）：

```
❌ 固定模板：这张表属于哪个主题域？（用户需从零回答）
✓ 动态生成：知识库显示 'credit_contract' 属于授信域，
  但此次粒度从「客户」改为「授信合同」，授信域已覆盖的表
  都没有此粒度，需要新建吗？
```

## 与现有表结构冲突时

当用户需求与 `schema/` 中已有表结构不一致时，立即指出冲突点，要求用户确认沿用已有结构还是新建/修改。
