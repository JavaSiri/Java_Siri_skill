# grill-with-docs

## 强制确认门

> **AI 绝对禁止跳过本 skill。** 无论用户是否认为口径已清晰、无论 AI 主观判断是否"够明确"，都必须执行本 skill 的追问流程，向用户逐指标发出确认提示，等待用户响应后才能推进。

跳过本 skill 即违反工作流纪律，视为未完成口径澄清。

在抛出任何口径问题前，必须先回答三个问题：

1. 当前正在处理哪个指标？（回答指标 `name`）
2. 该指标的 JSON status 是什么？（回答 "pending" 或 "confirmed"）
3. 上一个指标是否已标记 confirmed？未 confirmed 不得进入下一个

这三个问题回答后，方可发送追问消息。

## 追问纪律

- **禁止自主判断口径清晰**：无论 AI 认为口径多么清晰，都必须向用户发出确认提示，等待用户确认后方可标记为已澄清。AI 无权单方面跳过此步骤。
- **一次只问一个问题**：每次只抛出一个歧义点，等待用户回答后再问下一个。禁止一次性抛出多个问题。
- 必须在当前指标的口径完全澄清后，立即创建指标卡片并写入 `.cursor/knowledge/metrics/` 目录，方可进入下一个追问。

## 澄清结论记录

每澄清一个指标后，按格式写入 `.cursor/knowledge/metrics/{序号}_{指标技术名}.md`（具体格式参见 [METRIC-FORMAT.md](METRIC-FORMAT.md)），并在 `metrics-index.md` 中建立指标↔报表映射。

## 歧义收敛

当所有歧义点收敛后：
1. 确认 `metrics/` 目录下所有新指标均已生成卡片
2. 更新 `metrics-index.md`
3. 归档原始需求至 `.cursor/knowledge/requirement/`

## 与现有口径冲突时

当用户使用的指标与 `schema/` 或 `metrics/` 中已有定义不一致时，立即指出冲突点，要求用户确认沿用已有口径还是新建。

## 追问策略

### 挑战含糊表述

当用户使用含糊或被过度重载的业务术语时，提出精确的规范口径。

例如："你在说 '客户' —— 指的是 '注册客户' 还是 '有交易客户'？"

### 用数据场景压力测试

用具体的筛选条件和数据范围展开讨论，迫使双方精确界定边界：

- 某指标是否包含退款/撤销记录？
- 窗口期如何定义（自然日/交易日/自然月）？
- 关联维度的取值范围（历史全量 vs 近 N 天快照）？

### 与表结构交叉验证

当用户说明某指标的统计口径时，检查现有 DWD/DWS 表是否已支持。如果口径不一致，直接指出。

## 文件结构

```
.cursor/knowledge/
├── metrics/                  ← 指标卡片（口径沉淀）
│   ├── 001_授信通过数.md
│   └── ...
├── metrics-index.md          ← 指标↔报表映射索引
├── schema/                   ← 字段/表结构文档
├── requirement/              ← 原始需求存档
└── business/
    └── entity-relationship.md
```

## 谨慎建议创建 ADR

仅当同时满足以下三个条件时才提议创建 ADR（参见 [ADR-FORMAT.md](ADR-FORMAT.md)）：

1. 事后改变口径的代价可观（下游多个报表已按此口径开发）
2. 没有上下文会令人困惑（未来接手者会疑惑"为什么要这样定义"）
3. 确实存在多种口径定义，且有明确理由选择了其中一个

缺少任何一个条件，跳过 ADR。
